# CI / CD 의 기본 개념

CI / CD에서 CI는 **Continuous Integration** 즉 **지속적인 통합을 의미**하며, CD는 **Continuous Delivery** 즉 **지속적인 제공**, 혹은 Delivery 대신에 **Deployment**, **지속적인 배포로 의미**하기도 합니다.

CI / CD를 간단하게 말하면 애플리케이션 개발 단계부터 배포때 까지의 모든 단계를 자동화를 통해 효율적이고 빠르게 사용자에게 빈번하게 배포할 수 있도록 만드는 것이라고 할 수 있습니다.

빠르게 진화하고 변화하는 시대인 요즘에 어떻게 하면 시장과 고객의 요구에 빠르게 반응해서 제품을 출시, 업데이트 할 것인가에 대한 의문이 존재하고, 이 의문을 해결하기 위해서 수많은 기업이 CI / CD를 개발 프로세스로 사용하고 있습니다.

## CI (Continuous Integration) 지속적인 통합

버그 수정이나 새로 만든 기능들이 Main Repository에 주기적으로 빌드되고 테스트 돼서 Merge 되는 것을 말합니다.

요약하자면 **빌드 및 테스트 자동화** 과정입니다.

CI는 두 가지 특성을 생각하며 구현하면 좋습니다.

- 코드 변경 사항을 주기적으로 빈번하게 Merge해야 합니다.
  - 이는 두 명의 개발자가 있다고 예시를 들면 알기 편합니다. 오랫동안 서로 Merge를 하지 않고 같은 파일을 수정하다가 Merge를 하게 되면 서로 다른 코드를 어떻게 통합해서 적용해 나갈 건지 고생하게 됩니다. 이렇게 되면 새로운 기능을 개발하기 위해서 코드를 작성하는 것보다 Merge로 인한 충돌에 시간을 더 많이 할애 할 수 밖에 없는 구조가 됩니다. 그래서 어떻게 작은 단위로 나누어서 Main Repository에 반영하며, 배포할 수 있을지 생각하며 개발해야 하고 통합해나가야 합니다.
- 통합을 위한 단계 (Build, Test, Merge)의 자동화
  - 주기적으로 Merge 된 코드의 변경 사항이 자동으로 Build가 되어서 성공적으로 됐는지 확인이 되어야 하고, 새로 추가된 변경 사항뿐만 아니라 기존의 시스템의 다른 버그를 처리하지는 않았는지 Test까지도 자동으로 되어야 합니다.
  - 개발자들은 하루에도 몇 번씩 주기적으로 코드의 변경 사항을 Main Repository에 Merge 하는데, 이 과정에서 코드 리뷰가 잘 이뤄져서 코드가 적절한지 확인하는 절차가 필요합니다.
  - Merge가 되었다면 자동으로 팀에서 만든 CI 스크립트를 통해서 추가된 코드와 함께 Repository가 Build가 되고, Bulid 가 잘된다면 팀에서 작성한 여러 Test 스크립트가 실행되어 확인 작업에 들어가게 되고, 전부 통과가 된다면 배포가 되게끔 자연스럽게 이어지게 만들어야 합니다. 만약 Test에서 이상이 일어난다면 바로 경고가 떠서 개발자가 알 수 있게 알려야 합니다..

## CI의 장점

주기적으로 Merge를 하기 때문에 Merge 충돌을 피할 수 있어서 개발 생산성을 높일 수 있고, Merge 된 코드들은 자동으로 Build, Test가 되기 때문에 코드의 결함이나 문제점이 빠르게 발견될 수 있다는 장점이 있습니다. 또한 주기적으로 Merge 하기에 코드의 변경 사항이 적어, 버그를 수정하는 데 있어서 용이해집니다. 다 제쳐두고서라도 가장 최고의 장점이라고 할 수 있는 건 조금 더 나은 코드를 작성하게 되어 코드 질이 향상이 됩니다.

## CD (Continuous Delivery & Deployment) 지속적인 제공 or 지속적인 배포

CI를 통해서 주기적으로 Merge된 코드의 변경사항들이 자동으로 Build가 되고 Test가 되었다고 가정해보겠습니다.

## D에는 두가지의 경우가 나뉩니다.

### Delivery의 경우

배포하는 단계에서 배포할 준비과정인 Prepare Release를 거치고 준비된 Release가 아무런 문제가 없는지 Deploy Release 과정인 직접 개발자나 검증팀이 검증을 한 다음에 최종적으로 사용자에게 배포해도 되겠다라고 결정이 되면 **수동적으로 배포하는 단계**를 말합니다.

### Deployment의 경우

Release가 되자마자 자동으로 사용자에게 배포할 수 있도록 모든 것을 **자동화로 만드는 것**을 말합니다. 둘의 차이는 최종단계의 자동화 유무에 따라서 다르다고 생각하면 됩니다.

## CD의 장점

개발자에 입장에서 배포보다는 개발에 더욱 신경 쓸 수 있도록 도와주며, 원클릭으로 수작업 없이 Build, Test, Deploy까지의 자동화를 할 수 있어 능률을 올려준다는 장점이 있습니다.
